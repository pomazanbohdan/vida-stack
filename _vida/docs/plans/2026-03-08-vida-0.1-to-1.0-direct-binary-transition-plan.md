# VIDA 0.1 -> 1.0 Direct Binary Transition Plan

Purpose: decide whether VIDA should continue iteratively adapting the current Python/bash/`br` engine to the newly planned behavioral platform, or instead preserve `0.1` as the behavioral source of truth and implement the `1.0` binary directly in target format.

Status: approved architectural direction for framework planning and future spec cutting.

Date: 2026-03-08

---

## 1. Executive Decision

Yes: VIDA should **avoid** a long heavy adaptation of the current script runtime into the new behavioral platform.

Recommended direction:

1. keep `0.1` as the behavioral oracle and bugfix-speed bridge,
2. freeze semantics that matter,
3. write the `1.0` product directly in target binary form,
4. move only narrow bridge and export/parity surfaces through the old engine,
5. stop investing in large new long-horizon platform logic on top of `bash`/Python/`br`.

Hard rule:

1. do **not** mechanically port the current shell/Python stack one helper at a time into the new product shape,
2. do **not** keep reshaping the current engine to mimic the future instruction/state kernels,
3. do preserve and codify the runtime law, receipts, vocabulary, and gates that `1.0` must reproduce.

Compact formula:

`0.1 = semantic oracle + bridge + bugfix-speed runtime`

`1.0 = direct target implementation of state + memory + instruction + command + migration kernels`

## 2. Why This Decision Fits The Target

The target product direction is already explicit in `vida-stack`.

Source basis:

1. `vida-stack/README.md`
   - target is a self-hosted local binary,
   - current script runtime is a reference behavior layer,
   - the future system centers on state, memory, instruction, command, and migration kernels.
2. `vida-stack/VERSION-PLAN.md`
   - `0.1` remains behavioral source of truth,
   - `1.0` is a self-hosted local binary, not a daemon,
   - `0.2-0.9` exist to freeze semantics and define migration specs before the binary fully takes over.

Current source takeaways:

1. `README.md` explicitly says the repo is moving from script-based runtime into a local control binary and that markdown-first startup paths should leave the critical path.
2. `VERSION-PLAN.md` explicitly says `0.1` must remain the behavioral source of truth until the binary reproduces it.
3. Therefore the correct architecture move is not “keep evolving the script engine until it already looks like `1.0`”, but “freeze the behavior and implement the binary kernels directly”.

## 3. What The Current Engine Really Is

The current `_vida` runtime is already a strong behavioral reference system, but it is not the correct long-term product substrate.

Current shape:

1. shell command surface under `_vida/scripts/*.sh`,
2. Python engines for parsing, routing, scoring, validation, and state derivation,
3. `br` plus `.beads/issues.jsonl` plus queue-backed mutations for task state,
4. `.vida/logs/*` and `.vida/state/*` for receipts, telemetry, memory, run-graph, and diagnostics,
5. protocol-first runtime law in `AGENTS.md` and `_vida/docs/*`.

Local evidence:

1. `_vida/docs/script-runtime-architecture.md`
2. `_vida/docs/beads-protocol.md`
3. `_vida/docs/subagent-system-protocol.md`
4. `_vida/docs/agent-definition-protocol.md`
5. `_vida/docs/protocol-index.md`

Current engine strengths:

1. runtime law is already unusually explicit,
2. receipts, gates, verification, and fail-closed behavior already exist,
3. agent-definition work has already separated logic from prompt rendering,
4. orchestration, verification, approval, and diagnosis are already modeled as runtime surfaces.

Current engine bottlenecks:

1. task-state is still tightly coupled to `br`/JSONL queueing,
2. shell wrappers remain the canonical operator surface,
3. core orchestration is spread across many shell and Python helpers,
4. protocol docs still point directly to script entrypoints,
5. significant effort would be wasted if we keep reshaping this engine into the future binary instead of freezing and replacing it.

## 4. Architectural Decision Rule

Use this deterministic rule for future framework work.

### 4.1 Keep In 0.1 Only If

A change is allowed in the current runtime only if at least one is true:

1. it fixes correctness or stability of the current bugfix/delivery runtime,
2. it improves execution speed or reduces friction for current engineering work,
3. it is required to freeze behavior or export behavioral truth into parity artifacts,
4. it is required to keep the bridge to `1.0` viable,
5. it is a narrow defect fix in `br`/queue/log/receipt flow that blocks current productivity.

### 4.2 Move Directly To 1.0 If

A change should skip the old engine and be designed directly for the binary if at least one is true:

1. it introduces a new persistent runtime kernel,
2. it changes how instructions are stored, versioned, or rendered,
3. it replaces the task/state backend model,
4. it adds migration/versioning law,
5. it introduces durable command-first operator behavior intended to survive the script era,
6. it would otherwise require repeated shell/Python/`br` adaptation work.

### 4.3 Forbidden Middle Path

The following pattern is now forbidden:

1. invent a new behavioral surface in docs,
2. implement it deeply in shell/Python on top of `br`,
3. then later port it again into the binary.

If a behavior is future-kernel-grade, specify it now and implement it once in the target runtime.

## 5. What Must Stay In 0.1

`0.1` is still needed, but its mission becomes narrow and explicit.

### 5.1 Allowed Mission Of 0.1

1. current app/framework work continues to run on it,
2. bugfix throughput stays high,
3. semantic freeze artifacts are extracted from it,
4. parity tests and reference receipts are produced from it,
5. bridge wrappers or export helpers may be added when they make `1.0` implementation safer.

### 5.2 Required 0.1 Focus Areas

1. `br` and queue-backed mutation reliability,
2. boot/status/task snapshots,
3. route-receipt and run-graph correctness,
4. proof/verification artifact correctness,
5. instruction/runtime law clarity,
6. exportable semantic vocabulary for commands, states, receipts, approvals, memory kinds, and migration boundaries.

### 5.3 Explicit 0.1 Non-Goals

1. no major new persistence model work,
2. no long-horizon instruction storage engine,
3. no deep new shell orchestration layers intended to survive into `1.0`,
4. no large new behavioral platform features unless they are required for semantic freeze or active runtime correctness.

## 6. What Should Be Built Directly In 1.0

The following belongs in the target binary, not in late `0.1`.

### 6.1 State Kernel

Owns:

1. authoritative workflow/task state,
2. blockers, dependencies, receipts, review state,
3. compact resumability state,
4. one canonical mutation path.

Direct replacement target:

1. `br` task-state dependency,
2. `.beads/issues.jsonl`,
3. mutation queue wrappers,
4. current split between lifecycle SSOT and shell-level command glue.

### 6.2 Memory Kernel

Owns:

1. durable framework memory,
2. project memory foundation,
3. anomalies, lessons, corrections,
4. carry-over operational state.

### 6.3 Instruction Kernel

Owns:

1. runtime-stored core instructions,
2. versioned `Agent Definition` and `Instruction Contract` surfaces,
3. project/user overlays with validation,
4. effective instruction composition.

This is directly aligned with the already-created local research and runtime contract:

1. `_vida/docs/research/2026-03-08-agentic-agent-definition-system.md`
2. `_vida/docs/agent-definition-protocol.md`
3. `_vida/templates/instruction-contract.yaml`
4. `_vida/templates/prompt-template-config.yaml`

### 6.4 Command Kernel

Owns:

1. `vida boot`
2. `vida task ...`
3. `vida memory ...`
4. `vida status`
5. `vida doctor`

These commands should become the true operator surface instead of shell-first wrappers.

### 6.5 Migration Kernel

Owns:

1. schema migrations,
2. instruction migrations,
3. compatibility checks,
4. startup fail-closed behavior,
5. bridge from `0.1` artifacts/state into `1.0` runtime-owned storage.

## 7. Semantic Transfer Rule

The binary should inherit **semantics**, not **implementation accidents**.

### 7.1 Preserve Semantically

These behaviors must survive into `1.0`:

1. one SSOT for task/workflow state,
2. single-writer mutation law,
3. explicit receipts for route, escalation, approval, verification, and synthesis,
4. fail-closed authorization,
5. explicit writer ownership,
6. analysis -> writer -> coach -> verification law when required,
7. instruction behavior defined by allowlisted contracts, not implied prompt prose,
8. compact-safe resumability and deterministic handoff state.

### 7.2 Do Not Preserve Mechanically

These are implementation details and should not be ported 1:1:

1. shell wrapper layout,
2. Python helper boundaries,
3. `br --no-db` JSONL-safe mode,
4. `.beads/issues.jsonl` as the long-term backend,
5. direct CLI-driven subagent transport as the only orchestration transport.

## 8. Direct-Implementation Spec Set

Before large-scale coding in the target binary, the following spec artifacts should exist.

### Spec A. Semantic Freeze Spec

Purpose:

1. freeze command names,
2. freeze state vocabulary,
3. freeze receipt kinds,
4. freeze approval/review/memory/status enums,
5. define exact behaviors that `1.0` must reproduce.

Consumes:

1. `_vida/docs/protocol-index.md`
2. `_vida/docs/beads-protocol.md`
3. `_vida/docs/todo-protocol.md`
4. `_vida/docs/subagent-system-protocol.md`
5. `_vida/docs/agent-definition-protocol.md`

### Spec B. Command Tree Spec

Purpose:

1. define `vida` binary command tree,
2. map current command surfaces and wrappers into stable binary subcommands,
3. remove shell-first ambiguity.

### Spec C. State Kernel Schema Spec

Purpose:

1. define entities, receipts, statuses, dependencies, approval states, run state, and mutation rules,
2. replace `br`/JSONL with a binary-owned schema.

### Spec D. Instruction Kernel Spec

Purpose:

1. persist `Agent Definition`,
2. persist `Instruction Contract`,
3. persist `Prompt Template Configuration`,
4. define overlay precedence and compatibility validation.

### Spec E. Migration Kernel Spec

Purpose:

1. define what migrates from `0.1`,
2. define startup compatibility checks,
3. define rollback/fail-closed startup behavior,
4. define migration receipts.

### Spec F. Route And Receipt Spec

Purpose:

1. define route receipts,
2. define escalation receipts,
3. define verification and approval receipts,
4. define synthesis-ready state,
5. define machine-readable run-graph nodes.

### Spec G. Parity And Conformance Test Spec

Purpose:

1. verify that `1.0` reproduces frozen `0.1` semantics,
2. convert current runtime laws into executable binary conformance tests,
3. keep the port behavior-accurate instead of surface-accurate.

### Spec H. 0.1 Bridge Policy

Purpose:

1. define what old runtime work is still allowed,
2. define export/parity interfaces from `0.1` to `1.0`,
3. stop scope creep in the old engine.

## 9. Recommended Compact-Friendly Wave Order

This is the fastest safe order across several compact iterations.

### Wave 1. Semantic Freeze And Extraction

Goal:

1. extract the runtime law from the current engine,
2. freeze vocabulary and receipts,
3. stop uncontrolled behavioral drift in `0.1`.

Deliverables:

1. Semantic Freeze Spec,
2. 0.1 Bridge Policy,
3. source-registry and eval-plan gaps closed where still missing.

### Wave 2. Binary Migration Specs

Goal:

1. write the state, instruction, command, migration, and route/receipt specs,
2. make the binary buildable without further architectural debate.

Deliverables:

1. Command Tree Spec,
2. State Kernel Schema Spec,
3. Instruction Kernel Spec,
4. Migration Kernel Spec,
5. Route And Receipt Spec.

### Wave 3. Conformance Harness

Goal:

1. make `0.1` produce parity fixtures,
2. make `1.0` testable against them.

Deliverables:

1. Parity And Conformance Test Spec,
2. fixture/export rules,
3. first golden receipts and golden command outputs.

### Wave 4. Binary Foundation

Goal:

1. create the Rust workspace,
2. add embedded storage,
3. add startup version checks and migration runner,
4. expose minimal `vida` shell.

Deliverables:

1. minimal binary bootstrap,
2. version/migration check path,
3. command shell that can host later kernels.

### Wave 5. Kernels And Core Commands

Goal:

1. implement state, memory, instruction, and command kernels,
2. ship `vida boot/status/task/memory/doctor`,
3. begin self-hosting on selected flows.

### Wave 6. Switchover

Goal:

1. make VIDA use `vida` as primary operating surface,
2. demote script runtime to reference and fallback,
3. keep only thin compatibility shims where still required.

## 11. What The Current Runtime Should Still Optimize

The current engine should still be improved, but only under a narrow policy.

Allowed optimization themes:

1. `br` bottlenecks that slow all task creation and closure,
2. queue/lock/path correctness,
3. boot/status speed,
4. receipt correctness,
5. TODO/beads robustness,
6. proof/export surfaces needed for parity harness,
7. framework diagnostics that keep current development stable.

Not allowed as a default:

1. adding a second future-looking platform inside shell/Python,
2. building runtime-owned instruction storage on top of markdown-first startup,
3. deepening dependency on `br` if `1.0` will replace that backend,
4. re-solving binary-kernel problems in the old engine.

## 12. Risk Register

### Risk 1. Rewriting Too Early Without Freeze

Failure mode:

1. binary is implemented against moving semantics.

Mitigation:

1. semantic freeze before significant binary coding,
2. parity fixtures and conformance tests.

### Risk 2. Over-Investing In 0.1

Failure mode:

1. time is lost polishing the reference runtime into a pseudo-product that will still be replaced.

Mitigation:

1. enforce the `keep in 0.1 only if...` rule,
2. block future-kernel work from landing primarily in shell/Python.

### Risk 3. Mechanical Port Instead Of Semantic Port

Failure mode:

1. `1.0` reproduces script topology rather than runtime law.

Mitigation:

1. build against the semantic specs and parity tests,
2. use current scripts only as evidence sources, not as the target architecture.

### Risk 4. Bridge Breaks Current Productivity

Failure mode:

1. bugfix and framework work slow down while the binary is still immature.

Mitigation:

1. keep `0.1` stable and speed-focused,
2. scope current-runtime fixes to bridge productivity and correctness only.

## 15. Final Verdict

Architecturally, VIDA should take the faster and cleaner path:

1. stop trying to evolve the current engine into the final platform,
2. preserve `0.1` as the last reference runtime and operational bridge,
3. write the `1.0` binary around the target kernels directly,
4. use semantic freeze plus conformance specs to keep the rewrite exact,
5. optimize the old engine only where it preserves delivery speed, correctness, and migration confidence.
