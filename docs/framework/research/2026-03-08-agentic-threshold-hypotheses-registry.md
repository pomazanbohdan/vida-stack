# Agentic Threshold Hypotheses Registry

**Purpose:** Preserve candidate numeric or banded thresholds that emerged from research synthesis, without prematurely turning them into runtime law.

---

## Hypotheses

| ID | Threshold hypothesis | Why plausible | Current status |
|---|---|---|---|
| T01 | low-score bounded work should default to `single` | strongest research agreement supports single-lane validity for bounded work | candidate only |
| T02 | medium-score uncertain work often fits `dual` or `triad` | critique + proof value rises before quorum is justified | candidate only |
| T03 | quorum should be reserved for decomposable or conflict-heavy work | consensus overhead is only justified when extra perspectives matter | candidate only |
| T04 | arbiter mode should trigger before large fan-out on persistent disagreement | disagreement often signals ambiguity, not the need for more similar lanes | candidate only |
| T05 | prior weak effectiveness should raise proof burden before it raises agent count aggressively | more lanes without more proof can compound bad routing | candidate only |
| T06 | security-sensitive tasks should never rely on majority consensus alone | security decisions need veto or explicit approval logic | candidate only |
| T07 | volatile source families deserve the shortest freshness bucket | stale external guidance can silently invalidate plans | candidate only |
| T08 | route-regret and false-green metrics should act as retune triggers | these are strong signals of weak orchestration decisions | candidate only |

---

## Promotion Rule

No threshold here becomes runtime law until it is promoted through:

1. pilot or eval evidence,
2. protocol text,
3. test or health-gate support.
