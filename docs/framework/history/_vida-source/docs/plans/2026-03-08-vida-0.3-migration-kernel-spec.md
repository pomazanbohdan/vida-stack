# VIDA 0.3 Migration Kernel Spec

Purpose: define the authoritative migration model for direct `1.0`, freeze what `0.1` bridge/oracle surfaces may migrate into the binary, and separate migratable semantics from discardable topology and from the command, state, instruction, and route/receipt kernels.

Status: canonical `0.3` spec artifact for the direct `1.0` program.

Date: 2026-03-08

---

## 1. Executive Decision

The direct `1.0` migration kernel owns the semantic transition boundary between:

1. frozen `0.1` bridge/oracle surfaces, and
2. binary-owned `1.0` command, state, and instruction kernels.

The migration kernel freezes:

1. authoritative migration scope,
2. allowed migration inputs and bridge boundaries,
3. version and compatibility law across command/state/instruction migration inputs,
4. startup compatibility and fail-closed boot behavior,
5. migration states,
6. migration receipt families and proof obligations at the semantic level,
7. rollback notes and cutover preconditions.

The migration kernel does **not** own:

1. future root command-family law,
2. authoritative post-migration state entities,
3. authoritative instruction hierarchy or effective-composition law,
4. exact route/authorization/receipt payload schemas,
5. shell/Python/provider/CLI/storage topology.

Storage destination note:

1. migration targets embedded `SurrealDB` as the authoritative `1.0` substrate,
2. the canonical local backend is `kv-surrealkv`,
3. migration does not preserve `SQLite` or JSONL carriers as future product backends,
4. instruction-sidecar patches must be revalidated or rebound when bundled instruction versions change.

Compact rule:

`migrate frozen semantics through explicit compatibility law; reject topology leakage; fail closed when compatibility or proof is missing`

---

## 2. Why This Spec Comes Next

The command-tree spec already froze the future operator homes, the state-kernel spec already froze authoritative runtime facts, and the instruction-kernel spec already froze versioned instruction entities and behavior ownership.

The next blocker is the missing migration law that answers:

1. what `0.1` semantic truth may still cross into `1.0`,
2. what remains only a bridge/export carrier,
3. how compatibility is evaluated at boot,
4. how fail-closed startup and rollback posture work,
5. which migration proof surfaces exist before route/receipt schema work begins.

Without this artifact:

1. legacy topology could leak back into product law,
2. startup compatibility could remain implicit,
3. route/receipt work could accidentally absorb migration proof ownership,
4. cutover could be discussed without a canonical migration boundary.

## 4. Purpose Of The Migration Kernel Spec

This artifact answers one question:

1. what semantic compatibility and translation law must direct `1.0` own so that frozen `0.1` truth can be moved or validated without turning legacy carriers into product law?

This spec defines:

1. the authoritative migration scope from `0.1` bridge/oracle surfaces into direct `1.0`,
2. migratable semantics versus non-migratable topology,
3. version and compatibility law across command/state/instruction migration inputs,
4. startup compatibility and fail-closed boot law,
5. bridge import/export boundaries and allowed migration inputs,
6. migration states, receipt families, and proof obligations at the semantic level,
7. rollback notes and cutover preconditions,
8. the boundary between migration-owned behavior and command/state/instruction/route responsibilities,
9. migration-level invariants and non-goals.

This spec does not define:

1. exact receipt payload schemas,
2. exact route authorization law,
3. exact approval or verification payloads,
4. exact Rust module/storage layout,
5. provider- or CLI-specific startup transport,
6. full memory-kernel productization.

## 5. Authoritative Migration Scope From `0.1` Bridge/Oracle Surfaces

### 5.1 What Counts As Migratable Semantic Truth

The migration kernel may move or validate only frozen semantic truth from `0.1`, including:

1. bootstrap and activation law,
2. request-intent and tracked-flow law,
3. command-family meaning already frozen into `vida boot|task|memory|status|doctor`,
4. authoritative state vocabulary and mutation boundaries,
5. governance/readiness semantics,
6. run-graph and compact/resumability semantics,
7. instruction hierarchy, precedence, activation, and version-bearing boundaries,
8. canonical artifact-family meaning for receipts, proofs, context capsules, and parity fixtures,
9. durable framework-memory and diagnosis semantics where they affect future migration, doctor, or parity flows,
10. context-governance and WVP law when external validation affects migration readiness or proof.

Migration scope rule:

1. `0.1` remains the semantic oracle and bridge during the transition,
2. migration may consume its frozen semantic surfaces,
3. migration must not treat the `0.1` runtime substrate itself as the future product substrate.

### 5.2 What Remains Bridge-Only

The migration kernel may recognize bridge/export carriers, but it must keep them bridge-only:

1. shell/Python helper boundaries,
2. `br`, `.beads/issues.jsonl`, and JSONL-first modes,
3. legacy `SQLite`-backed helper/runtime assumptions,
4. `.vida/state/*`, `.vida/logs/*`, tmp-file choreography, and `.latest.*` conventions,
5. slash-command names, wrapper paths, helper verbs, shell profiles, and current entrypoints,
6. provider-specific prompt rendering, CLI dispatch wiring, cache/probe/output flags, and transport strings,
7. packet-rendering helpers, prompt file paths, and markdown-first startup choreography,
8. bridge/export helpers, normalizers, and compatibility shims used only to emit frozen examples or proofs.

Bridge-only rule:

1. a bridge carrier may be accepted as an input,
2. but it must never be promoted into `1.0` product law merely because it exists in `0.1`.

## 6. Migratable Semantics Versus Non-Migratable Topology

| Preserved semantic truth | Bridge-only carrier or discardable topology |
|---|---|
| future root command meaning and command-home boundaries | slash-command names, `_vida/commands/*.md`, shell entrypoints |
| canonical task/workflow state vocabulary and mutation law | `br`, `.beads`, queue choreography, JSONL-first storage, legacy `SQLite` carrier assumptions |
| instruction hierarchy, precedence, activation, and versioned entities | provider prompt text, template syntax, CLI/model/runtime bindings |
| receipt/proof artifact families as semantic categories | current file paths, tmp naming, `.latest.*`, helper-specific serialization |
| run-graph, context capsule, and resumability law | `.vida/state/run-graphs/*`, current capsule file paths, shell helpers |
| framework-memory and diagnosis semantics | current ledger paths and helper command spellings |

Negative-filter rule:

1. if a proposed migration rule depends on a current path, helper name, shell profile, provider transport, or `br` carrier as product law, reject it as topology leakage.

## 7. Version And Compatibility Law Across Command/State/Instruction Inputs

### 7.1 Required Compatibility-Bearing Input Families

The migration kernel must evaluate compatibility across these semantic input families:

1. `command_semantics`
   - the frozen future operator-family meaning and command-home boundaries,
2. `state_semantics`
   - the frozen authoritative state vocabulary, mutation boundaries, resumability, and blocker/readiness surfaces,
3. `instruction_semantics`
   - the explicit version-bearing instruction tuple:
     - `Agent Definition`
     - `Instruction Contract`
     - `Prompt Template Configuration`
     - validated overlay metadata where applicable,
4. `bridge_export`
   - the normalized bridge/export format that carries frozen semantic examples, fixtures, and importable artifacts,
5. `migration_kernel`
   - the currently selected migration law version that is interpreting the above families.

Explicit-version rule:

1. every required family must resolve to explicit version-bearing inputs before migration authorization,
2. exact manifest serialization remains open,
3. missing or unresolved version-bearing inputs are blocking.

### 7.2 Compatibility Evaluation Rules

Compatibility must be evaluated semantically, not topologically.

Rules:

1. `command_semantics` compatibility means legacy behavior can be mapped into the already frozen `vida boot|task|memory|status|doctor` surface without reviving old root names or shell families.
2. `state_semantics` compatibility means legacy carriers can translate into canonical `1.0` state entities and vocabularies without inventing new lifecycle enums or turning proofs into state facts.
3. `instruction_semantics` compatibility means version references resolve cleanly and `Instruction Contract` remains the canonical logic source during migration; `Prompt Template Configuration` must not become migration truth.
4. validated overlay input is compatibility-bearing only as input metadata; it must pass overlay schema validation before any migration binding proceeds.
5. bridge-export compatibility means the carrier is normalized enough to support semantic import without promoting the carrier topology itself.
6. compatibility may be classified per family for diagnosis, but normal boot execution remains blocked when any required family is unresolved or incompatible.

### 7.3 Compatibility Outcome Classes

The migration kernel may classify the evaluated input set as:

1. `compatible`
2. `compatible_with_migration_required`
3. `incompatible`
4. `insufficient_evidence`

Outcome rule:

1. only `compatible` and `compatible_with_migration_required` may proceed to normal boot flow,
2. `incompatible` and `insufficient_evidence` are fail-closed states.

## 8. Startup Compatibility And Fail-Closed Boot Law

### 8.1 Boot Preflight Order

`vida boot` is the primary migration preflight entry.

Before wider execution, boot must:

1. load framework invariants and validate overlay schema,
2. detect the allowed bridge inputs and resolve the source-version tuple,
3. evaluate command/state/instruction/bridge compatibility,
4. classify the current migration state,
5. stop and emit blocker visibility when compatibility or required proof is missing,
6. execute only deterministic, allowlisted migration paths when migration is required and authorized,
7. require migration verification proof before allowing ordinary execution to continue.

### 8.2 Fail-Closed Rules

Boot must fail closed when at least one is true:

1. a required version-bearing input family is missing or unresolved,
2. overlay schema validation fails,
3. legacy state cannot be translated into canonical `1.0` state entities,
4. instruction versions are incompatible or would invert the frozen hierarchy,
5. bridge inputs are present only as topology-heavy raw carriers without normalized semantic evidence,
6. required migration proof or rollback note is missing for a risky transition.

Hard rules:

1. boot must not silently widen into ad hoc repair logic,
2. boot must not fall back to transcript memory or ungoverned chat recap,
3. boot must not silently treat provider, CLI, helper, or path-level compatibility as semantic compatibility,
4. boot must not authorize normal task mutation while migration state is blocked or unverified.

### 8.3 Boot And Doctor Visibility Boundary

Operator visibility is split as follows:

1. `vida boot`
   - owns compact startup classification, go/no-go posture, and resumable next-step guidance,
2. `vida doctor`
   - owns detailed incompatibility diagnosis, blocker explanation, remediation guidance, and rollback visibility.

Boundary rule:

1. `boot` and `doctor` are the operator homes for migration visibility,
2. they must not become silent bypasses around migration, route, approval, or verification law.

## 9. Bridge Import/Export Boundaries And Allowed Migration Inputs

### 9.1 Allowed Migration Inputs

The migration kernel may import only normalized bridge inputs that carry frozen semantic truth, including:

1. golden command/output fixtures and command-family mapping evidence,
2. canonical state-transition examples, blocker/dependency/governance examples, run-graph examples, and context-capsule examples,
3. versioned instruction artifacts and validated overlay metadata,
4. canonical receipt-family examples and proof-bearing bridge artifacts,
5. framework-memory, diagnosis, and WVP examples when they are needed as semantic bridge evidence,
6. exact-vs-intentional delta registers and cutover-relevant parity references.

### 9.2 Allowed Bridge Exports From `0.1`

During the bridge era, `0.1` may continue exporting:

1. frozen semantic vocabularies,
2. canonical artifact-family examples,
3. parity fixtures,
4. compatibility helpers and normalizers,
5. reference proofs and rollback-relevant notes.

### 9.3 Forbidden Migration Inputs

The migration kernel must reject as authoritative migration truth:

1. raw chat memory or transcript recollection,
2. ungoverned log/path dumps used without normalized semantic classification,
3. current helper names, flags, profiles, or entrypoint spellings,
4. provider-specific prompt transport, CLI dispatch, or cache/probe details,
5. `br` topology and file-layout accidents treated as target-runtime requirements.

### 9.4 Import-Boundary Table

| Input family | Allowed as migration input | Target `1.0` owner | Not promoted as product law |
|---|---|---|---|
| command fixtures and operator examples | yes | command kernel plus parity harness | slash-command topology, wrapper paths |
| legacy state and resumability examples | yes | state kernel | `br`, `.beads`, `.vida/state/*` |
| versioned instruction artifacts | yes | instruction kernel plus migration validation | provider prompt/config transport |
| receipt/proof family examples | yes | route/receipt kernel later, migration proof now | file names, tmp choreography, helper serialization |
| framework-memory and diagnosis examples | yes, when needed for bridge evidence | later memory/doctor contracts | current ledger paths and helper spellings |
| parity/delta registers | yes | migration plus parity/conformance | ad hoc chat notes |

## 10. Migration States, Receipt Families, And Proof Obligations

### 10.1 Canonical Migration States

The migration kernel freezes these semantic migration states:

1. `no_migration_required`
   - compatibility is already satisfied and no bridge import is needed,
2. `bridge_input_required`
   - required normalized migration input is missing,
3. `compatibility_check_required`
   - inputs exist but compatibility has not yet been evaluated,
4. `migration_ready`
   - compatibility passed and an allowlisted migration path exists,
5. `migration_in_progress`
   - an authorized migration path is actively translating inputs,
6. `migration_blocked`
   - compatibility, proof, or required inputs failed,
7. `migration_applied_pending_verification`
   - migration ran, but verification proof is not yet satisfied,
8. `migration_verified`
   - migrated outputs are verified enough for normal execution,
9. `rollback_required`
   - the applied result cannot be trusted without rollback.

State rule:

1. these are migration-semantic states,
2. they do not redefine command families, task lifecycle, or route state.

### 10.2 Migration Receipt Families

At the semantic level, the migration kernel requires these receipt families:

1. `migration_compatibility_receipt`
   - records input families, resolved versions, compatibility outcome, and blockers,
2. `migration_application_receipt`
   - records the authorized migration path, source tuple, target tuple, and migrated semantic families,
3. `migration_verification_receipt`
   - records the proof that migrated outputs satisfy frozen semantics well enough for normal execution,
4. `migration_cutover_readiness_receipt`
   - records that migration-specific cutover gates are satisfied,
5. `migration_rollback_note`
   - records rollback boundary, trigger, and operator-visible consequences.

Schema boundary:

1. this artifact freezes the receipt families and their meaning,
2. later route/receipt work defines exact payload schemas where needed.

### 10.3 Migration Proof Obligations

Migration is not completion-ready unless proof exists for:

1. the explicit source-version tuple and target-version tuple,
2. the list of allowed bridge inputs actually consumed,
3. the semantic owner each migrated surface maps into,
4. the compatibility decision and any blocking reason,
5. the fact that legacy topology was not promoted into product law,
6. post-migration verification strong enough for normal execution,
7. rollback notes whenever the migration can affect future cutover or rollback safety.

Proof boundary:

1. migration owns the requirement that these proof families exist,
2. exact route/approval/verification payload schemas remain outside this spec.

## 11. Rollback Notes And Cutover Preconditions

### 11.1 Rollback Notes

Rollback notes are mandatory when migration touches authoritative state or instruction surfaces or changes cutover posture.

Every rollback note must state:

1. source tuple,
2. target tuple,
3. migrated semantic families,
4. safe rollback boundary,
5. non-reversible or manually reviewed surfaces,
6. operator-visible impact if rollback is required.

Rollback rule:

1. if a safe rollback path is unavailable for a risky migration, the migration must block rather than partially proceed.

### 11.2 Cutover Preconditions

The binary may become the primary operating surface only when all are true:

1. semantic freeze exists,
2. bridge policy exists,
3. command tree spec exists,
4. state kernel schema spec exists,
5. instruction kernel spec exists,
6. migration kernel spec exists,
7. route and receipt spec exists,
8. parity and conformance spec exists,
9. required bridge outputs and parity fixtures exist,
10. exact-vs-intentional delta rules exist,
11. startup fail-closed migration behavior exists,
12. required migration receipts and proofs exist,
13. operator-critical surfaces `vida boot|task|memory|status|doctor` are usable locally,
14. required conformance proofs exist.

Until those are true:

1. `0.1` remains the primary operating surface,
2. direct `1.0` remains under spec-first migration and productization.

## 12. Boundary Between Migration Semantics And Other Kernel Responsibilities

### 12.1 Command Kernel Boundary

The command kernel owns:

1. root command families,
2. command-home semantics,
3. operator-visible command boundaries.

The migration kernel owns only:

1. compatibility and migration behavior surfaced through `boot` and `doctor`,
2. fail-closed startup classification and migration status.

It must not:

1. redefine the five root families,
2. turn legacy shell families into future command law.

### 12.2 State Kernel Boundary

The state kernel owns:

1. authoritative `1.0` task/workflow entities and vocabularies,
2. dependency, blocker, resumability, governance, and reconciliation facts.

The migration kernel owns only:

1. translation of legacy carriers into those frozen state entities,
2. compatibility blocking when legacy state cannot map cleanly.

It must not:

1. redefine state vocabulary,
2. turn route proofs into state facts.

### 12.3 Instruction Kernel Boundary

The instruction kernel owns:

1. `Agent Definition`,
2. `Instruction Contract`,
3. `Prompt Template Configuration`,
4. precedence, activation, effective composition, and version-bearing boundaries.

The migration kernel owns only:

1. validation and movement of those version-bearing instruction surfaces at boot,
2. compatibility blocking when the hierarchy or versions cannot resolve lawfully.

It must not:

1. let `Prompt Template Configuration` become the migration truth source,
2. redefine instruction-owned fallback, escalation, output, or proof law.

### 12.4 Route And Receipt Kernel Boundary

The route/receipt kernel owns:

1. exact authorization law,
2. exact approval/escalation/verification payloads,
3. exact receipt schemas,
4. closure-ready proof artifacts.

The migration kernel owns only:

1. which migration receipt families must exist,
2. which compatibility and cutover proofs are required before normal execution or cutover.

It must not:

1. absorb exact route receipt schemas,
2. absorb exact approval or verification payloads,
3. absorb closure authorization law.

### 12.5 Memory And Doctor Boundary

This artifact may recognize framework-memory and diagnosis examples as bridge evidence, but:

1. it does not define the future memory kernel,
2. it does not define the full doctor/runtime contract,
3. it only defines the migration-facing compatibility and visibility boundary those later artifacts must respect.

## 13. Migration-Level Invariants

The direct `1.0` migration kernel must preserve these invariants:

1. migration preserves semantics, not topology,
2. `0.1` remains the oracle and bridge until cutover preconditions are met,
3. compatibility is explicit across command/state/instruction migration inputs,
4. missing, unresolved, or incompatible required inputs fail closed,
5. `vida boot` is the primary migration preflight entry,
6. `vida doctor` is the primary detailed migration-diagnosis surface,
7. allowed bridge inputs are normalized semantic carriers only,
8. `Instruction Contract` remains the canonical logic source during migration,
9. authoritative state facts remain separate from later route/receipt proofs during migration,
10. migration receipt families and rollback notes are required semantic artifacts, not optional logging,
11. undefined behavior remains forbidden by default,
12. scope must not widen silently into external future-scope domains.

---

## 14. Migration-Level Non-Goals

This spec does not:

1. redefine the frozen command tree,
2. redefine frozen state entities or vocabularies,
3. redefine the instruction hierarchy or precedence law,
4. define exact route/receipt schemas,
5. preserve current file paths, helper flags, shell profiles, or `br` carriers as product law,
6. define provider-specific prompt rendering or CLI transport behavior,
7. define exact Rust module/storage layout,
8. widen scope into `MCP`, `A2A`, `A2UI`, remote identity, gateways, or remote memory,
9. start Rust implementation work,
10. make `boot` or `doctor` a silent bypass around migration or route law.

---

## 15. Open Ambiguities

The following remain intentionally open:

1. the exact serialized manifest shape for the command/state/instruction compatibility tuple,
2. the exact multi-hop legacy-version support window versus a narrower bridge-only support window,
3. the exact storage shape and transition triggers for migration state inside the binary,
4. the exact payload schemas for migration receipts and cutover-readiness receipts,
5. the exact automation level of boot-applied migrations versus doctor-guided/manual migration paths,
6. whether framework-memory bridge import participates in the first cutover wave or a later memory-kernel wave,
7. the exact parity threshold required for `migration_verified` versus later full conformance closure.

Ambiguity rule:

1. later specs may refine these details,
2. they must not violate the migration boundary frozen here.
