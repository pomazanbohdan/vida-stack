# VIDA 0.3 DB-First Runtime Spec

Purpose: define that direct `1.0` runtime ownership is database-first, with filesystem artifacts retained only as export, bridge, or diagnostic surfaces.

Status: canonical `0.3` spec artifact for the direct `1.0` program.

Date: 2026-03-08

---

## 1. Executive Decision

Direct `1.0` runtime ownership is database-first.

Canonical rule:

1. authoritative runtime settings,
2. authoritative task/workflow state,
3. authoritative instruction artifacts,
4. authoritative project memory,
5. authoritative framework memory,
6. authoritative migration/version metadata,
7. authoritative operational configuration needed after install,

must live in the embedded `SurrealDB` runtime substrate.

Filesystem artifacts are allowed only as:

1. install-time bundled assets,
2. export surfaces,
3. bridge-era migration inputs,
4. bounded diagnostic traces.

Project-memory nuance:

1. project markdown/doc trees may synchronize with DB-owned project memory,
2. but they are not mandatory boot-owned substrate for the VIDA framework runtime itself.

Compact rule:

`after install, runtime truth lives in the DB; files are export or bridge surfaces only`

---

## 2. Binary-Only Product Goal

The desired operator posture is:

1. install binary,
2. run one boot/init command,
3. operate through the binary,
4. rely on DB-owned runtime surfaces,
5. export data or docs only when explicitly requested.

Direct `1.0` must not require day-to-day runtime dependence on loose `.vida/state/*`, `.vida/logs/*`, or `.beads/*` artifacts as canonical truth stores.

Binary build posture:

1. release binaries should be compiled with architecture-aware optimization for the intended target,
2. build/distribution surfaces must support per-architecture optimized binaries rather than one minimally tuned generic build only,
3. optimization policy may vary by target family (`x86_64`, `aarch64`, wasm) but must remain deterministic and explicit in build artifacts,
4. runtime correctness must not depend on one specific CPU extension being present unless that binary artifact is labeled for that architecture class.

---

## 3. What Must Be DB-Owned

The DB must own at minimum:

1. runtime settings needed by the binary after install,
2. `workflow_runtime` state for epics, tasks, dependencies, blockers, and execution plans,
3. execution telemetry state,
4. run-graph/resumability state,
5. governance/review state,
6. project memory,
7. framework memory,
8. instruction memory,
9. instruction sidecars,
10. `subagent_runtime` state for scorecards, strategy, eval/trace posture, and lifecycle status,
11. migration/version metadata,
12. export metadata and receipts.

Source-of-truth nuance:

1. human-edited source artifacts may live in Git source trees,
2. but the binary must ingest them into the DB before runtime use,
3. the DB remains the authoritative runtime mirror after ingest.

---

## 4. What May Remain Outside The DB

Outside-the-DB artifacts are allowed only for:

1. product-bundled immutable assets shipped with the binary,
2. import sources during migration,
3. explicit user-triggered exports,
4. temporary debug traces,
5. parity/proof artifacts that are exported from canonical DB state.

These are not SSOT runtime stores.

---

## 5. Bridge-Only Legacy Surfaces

The following remain bridge-only or export-only in direct `1.0`:

1. `.beads/issues.jsonl`,
2. `.vida/state/*`,
3. `.vida/logs/*`,
4. shell helper choreography,
5. loose file-ledger runtime memory.

They may exist during transition, but product law must treat them as non-authoritative.

---

## 6. Startup Rule

On boot the binary must:

1. open the embedded DB,
2. check runtime schema versions,
3. check bundled artifact versions,
4. run required DB migrations,
5. load immutable bundled instruction/config assets into the DB when needed,
6. fail closed on incompatible or partial migration state.

Hashing/runtime utility rule:

1. fast content/integrity hashing used for artifact ingest, projection anchors, and runtime receipts may use an architecture-optimized cryptographic hash implementation,
2. the current canonical fast hash choice is `BLAKE3`,
3. the binary may take advantage of architecture-specific SIMD/runtime detection when the chosen hash implementation supports it,
4. a later higher-precedence spec may replace the hash choice only through an explicit recorded library-evaluation decision.

---

## 7. Export Rule

All user-visible JSON, docs, memory dumps, logs, or parity bundles must be exportable from canonical DB-owned state.

Export rule:

1. exports are projections,
2. exports are not the runtime source of truth,
3. exported files may be re-importable only through explicit migration/import contracts.

---

## 8. Immediate Follow-Through

This spec requires:

1. an export protocol for JSON/docs/memory/logs,
2. migration of remaining runtime-file SSOT assumptions into DB-owned models,
3. implementation work to move boot/runtime configuration into the DB-owned substrate.
